# 🚀 100% 로컬 자원 RAG 콘텐츠 추천 시스템 PoC 데이터 파이프라인

온프레미스/오프라인 환경에서 데이터 적재부터 유사도 검색, 최종 추론까지의 전 과정을 성공적으로 검증한 데이터 파이프라인입니다.

---

# 📌 1단계: 로컬 임베딩 및 데이터 인제스천 (Data Ingestion)

로컬 메모리상의 더미 콘텐츠 데이터를 다국어 임베딩 모델을 활용해 고차원 벡터로 변환하고  
로컬 벡터 스토어에 적재하는 단계입니다.

```text
[로컬 더미 데이터]
        ↓
[HuggingFace 로컬 임베딩]
        ↓
[Chroma Vector DB (In-Memory)]
```

## 동작 상세

- 메모리에 정의된 `rag_test_documents`를 읽어들입니다.
- Nomic Embed v2 MoE 및 BGE-M3 모델을 로컬 자원(GPU/CPU)을 활용해 인메모리 상에서 실시간 임베딩합니다.
- 변환된 고차원 벡터를 경량 벡터 스토어인 ChromaDB에 순식간에 인덱싱합니다.

## 성능 지표 (DB 적재 속도)

| 모델 | 평균 소요 시간 | 특징 |
|---|---|---|
| BGE-M3 | 0.6 ~ 0.9초 | 우수한 한글/다국어 형태소 처리 속도 |
| Nomic-MoE-v2 | 1.3 ~ 1.9초 | MoE 구조 특유의 심층 게이팅 연산 소요 |

---

# 📌 2단계: 맥락 검색 및 리트리버 (Retrieval)

사용자의 자연어 질문 의도를 정밀하게 파악하여,  
벡터 DB에서 수학적으로 가장 유사도가 높은 관련 콘텐츠 1개(k=1)를 빠르게 찾아내는 단계입니다.

```text
[사용자 쿼리]
        ↓
[질문 벡터화]
        ↓
[유사도 비교 (Cosine Similarity)]
        ↓
[최적의 Context 추출]
```

## 동작 상세

- 사용자의 입력  
  `"나는 지금 퇴근하고 오자마자 재밌고 가벼운 컨텐츠를 보고 싶어."`  
  를 1단계와 동일한 임베딩 모델로 벡터화합니다.
- ChromaDB가 코사인 유사도 연산을 거쳐 가장 관련 깊은 문서 하나를 반환합니다.

## 모델별 검색 특성 검증 결과

| 모델 | 검색 결과 특성 | 평균 소요 시간 |
|---|---|---|
| Nomic-MoE-v2 | 감성적·추상적 맥락에 반응하여 '플리 채널' ASMR 콘텐츠 검출 | 0.3 ~ 0.4초 |
| BGE-M3 | '퇴근'이라는 직관적인 상황적 키워드 정합성을 포착하여 '유루캠' 힐링 애니메이션 검출 | 0.08 ~ 0.1초 |

# 📌 3단계: 가혹 조건 대응 및 프롬프트 합성 (Prompt Engineering)

검색된 문맥(Context)과 사용자 질문을  
LLM이 한눈에 파악하기 좋게 가공하고,  
매개변수가 작은 경량 LLM이 오작동하지 않도록 프롬프트 장벽을 세우는 단계입니다.

```text
[Context + User Query]
        ↓
[들여쓰기(Whitespace) 공백 100% 제거]
        ↓
[정제된 프롬프트 완성]
```

### 시행착오와 솔루션 (들여쓰기 붕괴 극복)

#### 문제점

파이썬의 삼중 따옴표 내부 들여쓰기 공백이 프롬프트에 유입될 경우,  
`qwen3.5:2b`와 같은 경량 모델이 지시사항이 아닌 잘못된 데이터 구조로 인지해  
공백을 반복 출력하는 현상이 발생했습니다.

#### 해결책

줄 맨 앞의 모든 공백(Tab/Space)을 완벽히 제거하여 Clean Template 공식을 완성했습니다.

# 📌 4단계: 로컬 LLM 추론 및 예외 처리 가드 (Generation & Safe Guard)

로컬에 탑재된 오픈소스 LLM을 사용해,  
획득한 문맥에만 기반한 신뢰도 높은 추천 사유를 완성하는 단계입니다.

```text
[정제된 프롬프트]
        ↓
[Ollama API 호출 (SDK 직접 연동)]
        ↓
[Thinking / Response 분리 파싱]
        ↓
[최종 출력]
```

## 시행착오와 솔루션 (Thinking Buffer 포화 현상 종결)

### 문제점

Qwen 추론 모델이 생각을 길게 전개하다가  
설정해 둔 최대 토큰 수(`num_predict=256`)를 모두 사용해버리며  
실제 답변 본문(`response=''`)은 출력하지 못한 채 응답이 비정상적으로 지연되는 문제가 발생했습니다.

최악의 경우 약 14분(853초) 동안 응답이 멈추는 병목 현상이 발생했습니다.

### 해결책 (방어 코드 탑재)

- LangChain 추상화 레이어를 제거하고 Ollama Python SDK를 직접 호출하여 지연 요소를 제거했습니다.
- 답변 출력단에 예외 가드 루틴을 설계했습니다.
- 최종 답변이 비어 있고 thinking 버퍼만 존재할 경우:
  - thinking 내용을 자동으로 최종 추천 사유로 전환하는 Fallback 파이프라인을 구축했습니다.

## 최종 성능 결과

| 항목 | 결과 |
|---|---|
| 기존 최악 응답 시간 | 약 14분 (853초) |
| 개선 후 평균 응답 시간 | 약 8 ~ 10초 |
| 실행 환경 | Apple Silicon M1 Pro GPU 가속 |
| 안정성 | 무조건 최종 추천 사유 출력 보장 |
