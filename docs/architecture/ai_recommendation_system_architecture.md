
# 퇴근 후 콘텐츠 추천 시스템 아키텍처 (Architecture Specification)

본 문서는 사용자의 기분, 가용 시간, 선호 플랫폼에 맞추어 퇴근 후 볼만한 콘텐츠(유튜브, 라프텔 등)를 RAG(검색 증강 생성) 기반으로 추천해 주는 시스템의 아키텍처 설계서입니다.

---

## 1. 시스템 아키텍처 다이어그램 (Concept)

```
[ 데이터 적재 (Data Ingestion) ]
1. 유튜브/라프텔 데이터 수집 및 정형 DB(SQLite) 저장
2. 텍스트 전처리 및 청킹 (text_processing.md)
3. 텍스트 임베딩 추출 (text_embedding_extraction.md)
4. 벡터 데이터 및 메타데이터 벡터 DB(ChromaDB) 저장

[ 실시간 추천 단계 (RAG Inference) ]
사용자 질문 ──> `FastAPI 서버` ──> 질문 임베딩 ──> `벡터 DB 검색 (Retrieval)` ──> 검색된 ID로 `SQLite 상세 조회` ──> `LLM (Generation)` ──> 추천 답변 제공
```

---

## 2. 세부 기술 스택 (Tech Stack)

### 🛠️ 개발 언어 및 환경
* **언어:** `Python 3.11+`
  * AI/LLM 프레임워크와의 최적의 호환성 제공.
* **패키지 매니저:** `Poetry`
  * 의존성 관리 및 가상환경 격리.

### 🏗️ 서버 아키텍처 (Backend)
* **메인 프레임워크:** `FastAPI`
  * 비동기(`async/await`) 처리를 통한 높은 동시성 지원.
  * `Pydantic`을 활용한 엄격한 데이터 유효성 검증.
* **RAG 오케스트레이션:** `LangChain` 또는 `LlamaIndex`
  * 검색-추출-생성 파이프라인의 연동 및 관리.

### 🤖 AI & LLM 모델
* **임베딩 모델:** `현재 미정`
  * 저비용, 고성능의 텍스트-벡터 변환 모델.
* **생성 모델 (LLM):** `미정`
  * 가성비가 뛰어나며 맥락을 이해한 자연스러운 추천 코멘트 작성에 특화.

## 3. 데이터 저장소 역할 분담 (Storage Role Division)

우리 시스템은 **정형 DB(SQLite)와 벡터 DB(ChromaDB)를 함께 사용**합니다. 각 데이터베이스의 역할을 명확히 정의하여 데이터 흐름의 혼선을 방지합니다. 이는 **포인터(Pointer) 패턴과 하이브리드(Hybrid) 패턴을 혼합**한 실용적인 접근 방식입니다.

### 3.1. SQLite (RDBMS): 진실의 원천 (Source of Truth)
* **역할:** 모든 콘텐츠의 **마스터(Master) 데이터를 저장**하고 관리합니다.
* **저장 데이터:**
    * 전체 콘텐츠의 상세 정보 (제목, 전체 줄거리, 제작사, 썸네일 URL 등)
    * 사용자 정보 (시청 기록, 북마크, 평점 등)
    * 관계형 데이터 (예: 시리즈와 에피소드 관계)
* **특징:** 데이터의 일관성과 무결성을 보장하며, 정교한 조인(JOIN) 쿼리가 가능합니다.

### 3.2. ChromaDB (Vector DB): 빠른 검색을 위한 인덱스 (Search Index)
* **역할:** 텍스트의 의미적 유사도를 기반으로 **관련성 높은 콘텐츠 청크를 빠르게 검색**하는 데 특화된 인덱스입니다.
* **저장 데이터:**
    * `embedding`: 검색의 기준이 되는 숫자 벡터.
    * `chunk_id`: 문서의 고유 ID.
    * `content_code`: RDBMS의 마스터 데이터와 연결하기 위한 **외래 키(Foreign Key) 역할**.
    * `text_content`: 검색 후 LLM에게 즉시 컨텍스트로 제공할 짧은 텍스트 청크.
    * `metadata`: 검색 시 필터링 조건으로 사용될 최소한의 정보 (플랫폼, 장르 등).
* **특징:** RDBMS에서 `LIKE` 검색이나 `Full-Text Search`로 찾기 힘든 **"의미 기반 검색"**을 담당합니다.

## 4. 핵심 파이프라인 동작 흐름

### A. 데이터 적재 파이프라인 (Data Ingestion)
1. **데이터 수집 및 마스터 저장:** 유튜브 API, 라프텔 크롤링 등을 통해 추천할 콘텐츠 정보를 수집하여 SQLite에 저장합니다.
2. **전처리 및 청킹:** 수집된 텍스트를 RAG에 적합하게 공백/태그를 제거하고 작은 단위(Chunk)로 분할합니다. (상세: `text_processing.md`)
3. **임베딩 변환:** 분할된 청크 텍스트를 임베딩 모델을 통해 숫자 배열(벡터)로 변환합니다. (상세: `text_embedding_extraction.md`)
4. **벡터 DB 저장:** 추출된 벡터 데이터, 청크 원본 텍스트, 그리고 SQLite 데이터와 연결될 `content_code`를 묶어서 ChromaDB에 저장합니다.

### B. 실시간 추천 파이프라인 (RAG Inference)
1. **사용자 요청:** 사용자가 질문(`"오늘 회사에서 힘들었어. 머리 식히며 볼 애니 추천해줘"`)을 입력합니다.
2. **질문 임베딩:** 입력된 질문을 임베딩 모델을 사용해 벡터로 변환합니다.
3. **유사도 검색 (Vector DB):** 변환된 질문 벡터를 바탕으로 ChromaDB에서 가장 유사도가 높은 상위 N개의 콘텐츠 청크와 `content_code`를 추출합니다.
4. **마스터 데이터 조회 (RDBMS):** 추출된 `content_code`를 사용하여 SQLite에서 해당 콘텐츠의 상세 메타데이터(썸네일, 제목 등)를 가져옵니다.
5. **프롬프트 합성:** 사용자 질문, 검색된 텍스트 청크, 그리고 상세 메타데이터를 결합하여 프롬프트를 생성합니다.
6. **답변 생성 (Generation):** LLM 모델이 최종 추천 코멘트와 콘텐츠 리스트를 생성하여 사용자에게 반환합니다.

## 5. 인프라 권장 사양 (사용자 1인 기준)

사용자 1명을 위한 개인 프로젝트이므로, 매우 가볍고 저렴한 서버 환경에서 구동 가능합니다.

* **CPU:** 1 vCPU
* **RAM:** 1GB ~ 2GB
* **Storage:** SSD 10GB ~ 20GB
* **추천 환경:** 오라클 클라우드 프리 티어, AWS EC2 프리 티어(`t3.micro` 등)
